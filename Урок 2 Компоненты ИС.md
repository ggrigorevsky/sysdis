### Балансировщик нагрузки (load balancer)

**Проблема:** Один сервер не справляется с трафиком. Если он упадет — система умрет.
**Решение:** Распределение входящих запросов между несколькими серверами (бэкендами).

#### Типы:
- **L4 (Транспортный уровень):** Работает с TCP/UDP, не смотрит на содержимое запроса. Быстро, но "слепой" к HTTP-методам.
- **L7 (Прикладной уровень):** Понимает HTTP/HTTPS. Может маршрутизировать запросы на основе URL (например, `/api/*` → одни серверы, `/static/*` → другие).

#### Основные алгоритмы:

- Round Robin (самый простой и популярный: по кругу)
- Weighted RR (по кругу с весами)
- Least Connections (куда меньше занято)
- Least Response Time (где наименьшее время ответа)
- IP Hash (для привязки сессий, sticky session)

##### Round Robin 
**Механизм:** Балансировщик хранит указатель на последнего выбранного сервера. При каждом новом запросе указатель сдвигается на следующего по списку сервера. Дойдя до конца списка, возвращается к началу.

**Как это работает:** Запросы распределяются строго по очереди: 1-й → сервер A, 2-й → сервер B, 3-й → сервер C, 4-й → снова A.

##### Weighted RR
**Механизм:** Каждому серверу присваивается вес (например, A — вес 5, B — вес 1).  Обычно вес зависит от "железа" сервера: больший вес присваивается более мощному серверу. Балансировщик составляет виртуальную последовательность, где сервер A повторяется 5 раз, а сервер B — 1 раз. Затем применяет обычный Round Robin к этой последовательности.

**Как это работает:** Если сервер мощнее — он получает пропорционально больше запросов. При весах 5 и 1 на каждые 6 запросов первый сервер обработает 5, второй — 1.

##### Least Connections
**Механизм:** Запрос направляется серверу, у которого в данный момент меньше всего активных соединений.
 
**Как работает:** Балансировщик ведёт счётчик открытых соединений на каждом сервере.
 
**Идеально для:** Долгих сессий (например, загрузка файлов, потоковое видео, API с разным временем ответа).

##### Least Response Time
**Механизм:** Отправляет запрос серверу, который одновременно имеет меньше всего активных соединений и наименьшее среднее время ответа.

**Как работает:** Балансировщик хранит и постоянно перерассчитывает экспоненциально взвешенное скользящее среднее время ответа и отправляет запрос на сервер, у которого оно наименьшее.

**Минус:** Требует постоянного мониторинга задержек, что создаёт небольшие накладные расходы.

##### IP Hash
**Механизм:** Балансировщик вычисляет хэш от IP-адреса клиента (или его id) и направляет запрос на определённый по этому хэшу сервер.

**Как работает:** Балансировщик вычисляет хэш от IP-адреса клиента (например, CRC32 или MD5), затем берёт остаток от деления этого хэша на количество серверов в пуле. Полученное число определяет, какой сервер из упорядоченного списка получит запрос. Поскольку хэш от одного и того же IP всегда одинаков, а список серверов не меняется, все запросы одного клиента попадают на один и тот же сервер.

**Цель:** Обеспечить «липкость» (sticky session) — чтобы запросы одного пользователя всегда попадали на один и тот же сервер (нужно для хранения сессий в локальной памяти).

**Проблема:** При добавлении/удалении сервера хэш-распределение ломается (проблема решается консистентным хэшированием, о котором мы поговорим в отдельном уроке). 

#### Другие алгоритмы:
Существуют и другие более экзотические алгоритмы балансировки нагрузки, например Dynamic Load Balancing или Power of Two Choices. Их крайне редко спрашивают на интервью, поэтому не будем на них останавливаться. Если интересно, можно почитать про них самостоятельно.

#### Примеры балансировщиков:
* Nginx
* HAProxy, 
* AWS ELB, 
* Istio

> _Ключевая идея:_ Балансировщик должен сам быть отказоустойчивым (Active-Passive или Active-Active кластер).

### Rate Limiter (Лимитер запросов)

**Проблема:** Злоумышленник может положить систему DDoS-атакой, отправляя миллионы запросов в секунду. Но rate limiting решает и более широкий класс проблем: непреднамеренные перегрузки (баг в клиенте с бесконечным циклом), монетизацию (разные лимиты для free/paid тарифов) и честное распределение ресурсов между пользователями.

**Решение:** Ограничение количества запросов от одного пользователя/IP/API-ключа за интервал времени.

#### Популярные алгоритмы (стратегии):
1. **Token Bucket**
2. **Leaky Bucket**
3. **Fixed Window Counter**
4. **Sliding Window Log**
5. **Sliding Window Counter**
6. **GCRA**

#### По какому признаку лимитировать

На интервью стоит уточнить этот вопрос первым делом — от ответа зависит вся архитектура.

- **По IP** — простейший вариант, но ненадёжен: NAT объединяет тысячи пользователей за одним IP, а злоумышленник легко меняет адрес.
- **По User ID / API-ключу** — точнее, но требует аутентификации до проверки лимита.
- **По эндпоинту** — `POST /login` лимитируется жёстче, чем `GET /catalog`.
- **Комбинация** — например, `user_id + endpoint` для разных лимитов на разные операции.

> _На интервью:_ «Лимитируем по IP или по авторизованному пользователю?» — задайте этот вопрос явно.

#### 1. Token Bucket («Ведро с токенами»)

**Как работает:**  
У вас есть «ведро» (счетчик токенов) фиксированной вместимости (burst capacity). Токены добавляются в ведро (инкеремент счетчика) с постоянной скоростью (например, 10 токенов/сек). Каждый запрос забирает (декремент) 1 токен. Если токенов нет — запрос отклоняется (429).

##### Плюсы
- ✅ **Гибкость:** Разрешает кратковременные всплески (bursts) — до полного объема ведра.
- ✅ **Легко настраивать:** Можно менять скорость пополнения и размер ведра независимо.
- ✅ **Популярность:** Поддерживается из коробки в nginx (`limit_req`), Redis, Guava (Java), Bucket4j.
- ✅ **Не блокирует надолго:** Запрос либо выполняется сразу, либо отклоняется (без очереди).

##### Минусы
- ❌ **Burst может перегрузить backend:** Если ведро полное, клиент может мгновенно выстрелить всеми накопленными токенами (в рамках размера ведра). Backend получит пиковую нагрузку.
- ❌ **Сложность синхронизации:** Распределенные системы требуют осторожной работы с shared state (обычно Redis + Lua).
- ❌ **Не хранит историю:** Нельзя точно ответить, когда именно был предыдущий запрос.

---

#### 2. Leaky Bucket («Дырявое ведро»)

**Как работает:**  
Запросы поступают в очередь фиксированного размера. Обработчик забирает запросы из очереди с постоянной скоростью. Если очередь заполнена — запросы отбрасываются.

##### Плюсы
- ✅ **Сглаживает трафик:** Идеально для защиты downstream-сервисов, которые не выдерживают резких всплесков (базы данных, старые SOAP-сервисы).
- ✅ **Постоянная выходная скорость:** Легко прогнозировать нагрузку.
- ✅ **Простое поведение:** Подходит для шейпинга трафика (traffic shaping), а не только для отбрасывания.

##### Минусы
- ❌ **Задержки:** Запрос может долго ждать в очереди.
- ❌ **Меньше гибкости:** Нельзя временно пропустить «пачку» запросов, даже если мощности позволяют.
- ❌ **Реализация сложнее:** Чистый Leaky Bucket с очередью редко используют в HTTP API — вместо него применяют Token Bucket или GCRA.

> Примечание: в реальных API «Leaky Bucket» часто путают с Token Bucket с фиксированной обработкой. Классический «Leaky Bucket» сегодня менее популярен.

---

#### 3. Fixed Window Counter («Фиксированное окно»)

**Как работает:**  
Время делится на фиксированные интервалы (например, по 1 минуте). Для каждого ключа хранится один счётчик, который сбрасывается в ноль в начале нового интервала. Если счетчик превышает порог, запросы отбрасываются.

**Плюсы:**
- ✅ O(1) память, тривиальная реализация, легко читать метрики
- ✅ Легко объяснить бизнесу: "100 запросов в минуту" — интуитивно понятное правило
- ✅ Счётчики сбрасываются предсказуемо — удобно для лимитов по тарифным планам (например, 10 000 запросов в месяц)

**Минусы:**
- ❌ Граничный эффект* делает его непригодным там, где важна точность
- ❌ Не защищает от кратковременных пиков внутри окна — можно отправить все 100 запросов за первую секунду минуты
- ❌ При распределённой системе счётчики на разных серверах нужно синхронизировать, иначе каждый сервер считает лимит независимо

\* - *Граничный эффект: Лимит - 100 запросов в минуту. Пользователь отправляет 100 запросов в 12:00:59 — они укладываются в окно 1. Затем в 12:01:01 отправляет ещё 100 запросов — они укладываются в окно 2. Оба окна видят по 100 запросов и считают лимит не превышенным, хотя фактически клиент отправил 200 запросов за 2 секунды.*

**Где применяется:** Внутренние сервисы, где допустима погрешность; базовая защита на уровне nginx.

---

#### 4. Sliding Window Log («Скользящее окно логирования»)

**Как работает:**  
Для каждого пользователя/ключа хранится список временных меток всех запросов за последние N секунд. При новом запросе удаляются метки старше окна, подсчитывается оставшееся количество запросов. Если счетчик превышает порог, запросы отбрасываются.

##### Плюсы
- ✅ **Максимальная точность:** Нет «ложных срабатываний» на границах временных интервалов (как у Fixed Window).
- ✅ **Честное ограничение:** Учитывает точный момент каждого запроса.
- ✅ **Хорошо подходит для малых окон:** Например, лимит 3 запроса в 5 секунд.

##### Минусы
- ❌ **Огромный расход памяти:** Нужно хранить timestamp каждого запроса. При 10k RPS на пользователя — катастрофа.
- ❌ **Медленно для больших окон:** При окне в 1 час и высокой частоте запросов хранение и удаление старых меток становятся дорогими.
- ❌ **Плохо для распределенных систем:** Синхронизация списков между нодами — головная боль.

**Где применяется:** Только в low-throughput или прототипах. В продакшене почти не используется без агрегации.

---

#### 5. Sliding Window Counter (более практичный вариант)

**Как работает:**  
Гибрид Fixed Window + скользящего веса.  
Разбиваем время на маленькие интервалы (например, 1 секунду). Храним счетчики для текущего и предыдущего интервалов. При запросе:

```
rate = prev_count * (1 - weight) + cur_count
```

где `weight` — доля прошедшего времени в текущем окне.

##### Плюсы

- ✅ **Экономно по памяти:** O(1) на пользователя (всего 2 счётчика).
- ✅ **Достаточно точно:** Ошибка ~1–5%, но для rate limiting это допустимо.
- ✅ **Быстро:** Никаких списков, только инкременты в Redis.
- ✅ **Подходит для распределенных систем:** Легко делать через атомарные операции.

##### Минусы
- ❌ **Не идеально точно:** В редких граничных случаях может пропустить чуть больше или меньше разрешенного.
- ❌ **Чувствительность к размеру интервала:** Если интервал 1 час, «скольжение» грубое.

**Где применяется:** Это стандарт для Redis-based ограничений (например, `redis-cell` с GCRA — близкий алгоритм). GitHub, Stripe используют похожие подходы.

---

#### 6. GCRA (Generic Cell Rate Algorithm) — «продвинутый Token Bucket»

**Как работает:**  
Алгоритм из телекома (ATM networks). Работает с «временем следующего разрешенного запроса» (TAT — Theoretical Arrival Time). Каждый запрос сдвигает TAT вперед, но ограничивает bursts.

##### Плюсы

- ✅ **Объединяет лучшее от Token Bucket и Leaky Bucket:** Можно разрешать bursts, но сохранять долгосрочную среднюю скорость.
- ✅ **Требует O(1) памяти:** Хранит только одно число (TAT) на ключ.
- ✅ **Атомарность:** Легко реализовать в Redis (одна команда `SETNX` + вычисления).
- ✅ **Отсутствие очереди:** Нет блокировок и задержек ожидания.

##### Минусы
- ❌ **Сложность понимания:** Менее интуитивен, чем Token Bucket.
- ❌ **Меньше библиотек:** Из коробки есть в `redis-cell` и некоторых Go-пакетах, но не везде.
- ❌ **Сложнее объяснить stakeholders:** Параметры TAT и burst tolerance менее интуитивны, чем «10 запросов в секунду, burst 50» у Token Bucket — на code review и в документации потребуется дополнительное объяснение.

| ---------------------- | ---------------- | -------- | ----------------- | -------------------- | ---------------- |

#### Что возвращать клиенту

**HTTP-статус:** `429 Too Many Requests`

**Стандартные заголовки ответа:**

| Заголовок | Значение |
| --- | --- |
| `X-RateLimit-Limit` | Максимально допустимое количество запросов |
| `X-RateLimit-Remaining` | Осталось запросов в текущем окне |
| `X-RateLimit-Reset` | Unix timestamp сброса счётчика |
| `Retry-After` | Через сколько секунд повторить запрос |

**Что делать клиенту:** Exponential backoff + jitter (случайный разброс), чтобы избежать «стадного эффекта» — когда тысячи клиентов одновременно получают 429 и синхронно штурмуют сервер через одинаковый промежуток времени.

---

#### Когда что выбирать?

- **Token Bucket** — для 90% REST API, когда нужны bursts и простота.  
    _Пример:_ Лимит 1000 запросов в минуту с возможностью 100 запросов за 1 секунду.
- **Sliding Window Counter** — для высокой нагрузки и требований к памяти.  
    _Пример:_ Миллионы пользователей, лимит 5 запросов/сек (Redis).
- **GCRA** — если важна идеальная плавность + bursts, память критична.  
    _Пример:_ Телеметрия, IoT, финансовые API.
- **Sliding Window Log** — только для очень низкого RPS (например, админка с 1 запросом в минуту).  
    _Пример:_ Лимит 3 попытки логина за 10 минут.
- **Fixed Window Counter** — только для внутренних сервисов, где граничный эффект некритичен.  
    _Пример:_ Внутренний cron-джоб, которому нельзя делать более 100 запросов к партнёрскому API в час.
- **Leaky Bucket** — почти не используйте. Вместо него возьмите Token Bucket с очередью обработки на клиенте.

---

#### Практический совет

В современных распределенных системах **лучшая комбинация** —  
**GCRA** (через `redis-cell`) или **Sliding Window Counter** (через Redis + Lua).  
Для монолита на одной ноде — **Token Bucket** in-memory (Guava, Bucket4j).

#### Где размещать

| Уровень | Плюсы | Минусы |
| --- | --- | --- |
| На клиенте | Снижает исходящий трафик | Ненадёжно — клиент контролирует сам себя |
| API Gateway / балансировщик | Централизованно, до вашего кода | Одна точка отказа |
| Отдельный сервис (sidecar) | Масштабируется независимо | Добавляет latency на каждый запрос |
| Внутри сервиса | Минимальная latency | Нельзя лимитировать между инстансами |

**Рекомендация:** API Gateway + Redis — оптимальный баланс для большинства систем.

> _Ключевая идея:_ Rate limiter должен быть распределённым — если у вас 10 балансировщиков, они должны синхронизировать счётчики через Redis, иначе каждый балансировщик будет считать лимиты независимо и фактический лимит вырастет в 10 раз.

###  Генератор ID (ID Generator)

#### Auto_increment

Представим, что нам необходимо генерировать id какой-то сущности в распределенной системе. Например id заказа на маркетплейсе. 
Для этих целей можно было бы использовать auto_increment в БД, но этот подход плохо работает при высоких нагрузках.
Auto_increment — это простой счётчик внутри одного инстанса БД. Пока система небольшая, он работает идеально. Но с ростом нагрузки проявляются три фундаментальных недостатка.

**1. Невозможность горизонтального масштабирования**  
Если нужно добавить второй инстанс БД (шардирование, реплика для записи), два независимых счётчика начнут генерировать одинаковые ID. Вставка пользователя с `id=1001` произойдёт на обоих шардах — возникнет коллизия. Частичное решение — задать разные шаги (`AUTO_INCREMENT_INCREMENT = 2`, `AUTO_INCREMENT_OFFSET = 1/2`), но это костыль: при добавлении третьего шарда придётся мигрировать всю схему.

**2. Единая точка отказа и bottleneck**  
Счётчик защищён глобальным мьютексом. При высоком параллелизме потоки выстраиваются в очередь за следующим значением. Кроме того, каждая вставка требует сброса WAL/redo-лога на диск (fsync). Практический потолок — около 10 000–30 000 INSERT/сек на один инстанс с SSD. При нагрузке выше этого значения база становится узким местом всей системы.

**3. Предсказуемость и утечка информации**  
Последовательные числовые ID позволяют конкурентам оценить объём базы: достаточно создать двух пользователей с интервалом в час и посмотреть разницу ID. Также ID легко перебираются: запрос `/api/orders/1`, `/api/orders/2` и т.д. открывает данные без дополнительной защиты.

#### Snowflake ID: архитектура решения

Twitter разработал Snowflake в 2010 году, когда столкнулся именно с этими проблемами при масштабировании ленты новостей. Идея: генерировать уникальные ID **децентрализованно**, без координации между узлами, но при этом сохранять монотонность (возрастающий порядок во времени).

Snowflake ID — это **64-битное целое число**, разбитое на три части:

```
|--- 41 бит ---|--- 10 бит ---|--- 12 бит ---|
   timestamp      machine ID     sequence
```

**Timestamp (41 бит)** — миллисекунды с произвольной эпохи (Twitter выбрал 2010-11-04). 41 бит хватает примерно на 69 лет. Монотонность гарантирована: более поздние ID всегда больше более ранних.

**Machine ID (10 бит)** — уникальный номер узла-генератора, 2¹⁰ = 1024 возможных узла. Каждый сервер получает свой номер при старте (из ZooKeeper, конфига или координатора). Именно это устраняет проблему коллизий: два разных узла никогда не выдадут одинаковый ID, даже в одну миллисекунду.

**Sequence (12 бит)** — счётчик внутри одной миллисекунды на одном узле, 2¹² = 4096 значений. Если узел генерирует более 4096 ID за миллисекунду, он ждёт следующей. Итоговый теоретический предел одного узла: **4 096 000 ID/сек**. Для кластера из 1024 узлов — свыше 4 миллиардов ID/сек.

### Как Snowflake решает проблемы auto_increment

|Проблема auto_increment|Решение в Snowflake|
|---|---|
|Коллизии при шардировании|Machine ID гарантирует уникальность без координации|
|Мьютекс и узкое место БД|Генерация на уровне приложения, без обращения к БД|
|Предсказуемость ID|ID не последователен глобально, но и не случаен — компромисс|
|Невозможность масштабирования|До 1024 независимых генераторов параллельно|

Монотонность также решает проблему UUID v4: случайные 128-битные UUID при вставке в B-tree индекс вызывают постоянные page splits*  и фрагментацию — база работает медленнее. Snowflake ID возрастают во времени, поэтому новые записи добавляются в конец индекса, как и у auto_increment.

\* - *Page split — это операция в B-tree индексе, при которой переполненная страница разбивается на две, чтобы освободить место для новой записи. Эта операция увеличивает время insert в БД.*

### Практическое применение

Snowflake используют Twitter, Discord, Instagram (с модификацией), Mastodon. Discord хранит в ID время отправки сообщения — это позволяет выбирать сообщения за диапазон дат без дополнительной колонки `created_at`.

**На собеседовании** при вопросе о генерации ID стоит упомянуть Snowflake как решение для высоконагруженных систем с горизонтальным масштабированием. Проблема в том, что на это дополнительно нужна небольшая инфраструктура для раздачи Machine ID узлам (ZooKeeper или аналог). Если этой инфраструктуры нет — проще взять **UUID v7** (тот же принцип монотонного timestamp + случайный суффикс, но без координации узлов).
Поэтому данный подход целесообразно применять не всегда, а только если нужны десятки тысяч ID в секунду или более.

> Генератор ID — типичный **функциональный сервис** без состояния или с легким состоянием (ZooKeeper для координации нод).

### Job Scheduler (Планировщик заданий)

**Проблема:** Нужно выполнять фоновые задачи: отправить email в 10 утра, ресайзить картинки раз в час, чистить логи.

**Решение:** Система для запуска задач по расписанию или по событию.

#### Архитектура:
- **Master-Worker:** Мастер хранит расписание в БД (или etcd), выдает задания воркерам.
- **Leader Election:** Несколько нод, одна — лидер (через ZooKeeper/Consul).

#### Важные концепты:

- **Гарантия выполнения задания в очереди**: **at-most-once** (почта) vs **at-least-once** (генерация отчета).
- **Idempotency** — задача должна быть безопасна для повторного выполнения.
- **Delayed Queue** — если задачу нужно выполнить через некоторое время

#### At-most-once и At-least-once

Это гарантии доставки сообщения или выполнения задачи. Они описывают, что произойдёт, если в процессе выполнения что-то пойдёт не так — упадёт сервер, оборвётся сеть, закончится память.

**At-most-once** — задача выполнится **не более одного раза**. Если что-то пошло не так до подтверждения выполнения — задача будет потеряна и повторно не запустится.

Планировщик отправляет задачу воркеру и сразу помечает её как выполненную, не дожидаясь результата. Если воркер упал в процессе — задача не перезапустится.

_Когда подходит:_ когда потеря допустима, а повтор опасен. Пример — отправка email-уведомления. Лучше не отправить письмо вовсе, чем отправить его дважды и получить жалобу на спам.

**At-least-once** — задача выполнится **не менее одного раза**. Если подтверждение не получено — задача перезапустится. При этом возможно выполнение более одного раза.

Планировщик отправляет задачу и ждёт подтверждения (ACK). Если воркер упал или не ответил в течение таймаута — задача ставится в очередь повторно.

_Когда подходит:_ когда потеря недопустима, а повтор безопасен. Пример — генерация финансового отчёта. Лучше сгенерировать его дважды, чем не сгенерировать вовсе.

**Exactly-once** — это гарантия того, что задача выполнится **ровно один раз**, даже при сбоях. Достичь **exactly-once** крайне сложно. 

В этих концепциях важно понимать, что на практике даже без дополнительных ухищрений в 99,9% случае задача будет выполняться ровно один раз. Речь в этих концепциях идет о крайних случаях, вызванных сбоями (упал сервис, БД или пропала сеть), а не о штатной работе системы. Мы еще раз вернемся к этой теме, когда будем разбирать работу kafka.

#### Idempotency (Идемпотентность)

**Идемпотентная операция** — это операция, которую можно выполнить любое количество раз, и результат будет тот же, что и при однократном выполнении.

Это свойство критически важно при использовании At-least-once гарантий: раз задача может выполниться несколько раз, она должна быть написана так, чтобы повторный запуск не ломал систему.

**Пример не идемпотентной задачи:**

```
Списать 1000 рублей со счёта пользователя
```

Если задача запустится дважды — спишется 2000 рублей. Это баг.

**Пример идемпотентной задачи:**

```
Установить статус заказа #4521 = "оплачен"
```

Сколько бы раз эта задача ни запустилась — результат одинаковый.

**Как сделать задачу идемпотентной:**

- Использовать `idempotency_key` — уникальный ключ задачи. Перед выполнением проверять: "была ли уже выполнена задача с этим ключом?". Если да — пропустить.
- Использовать `upsert` вместо `insert` в БД (вставить если нет, обновить если есть).
- Хранить состояние задачи и проверять его перед выполнением.

На практике At-least-once + Idempotency = надёжная система. Именно такую комбинацию используют Kafka, SQS, Celery и большинство брокеров сообщений.

#### Delayed Queue (Отложенная очередь)

Обычная очередь задач выполняет задачи сразу по мере поступления. Delayed Queue позволяет поставить задачу в очередь с условием: **выполнить не раньше чем через N минут/часов**.

**Типичные сценарии:**

- Отправить письмо-напоминание через 24 часа после регистрации
- Автоматически закрыть заказ, если оплата не поступила через 30 минут
- Повторить упавшую задачу через 5 минут (retry с задержкой)
- Запустить ночной отчёт в 03:00

**Как это реализуется:**

Задача сохраняется в хранилище с меткой `execute_at = now() + 30 минут`. Планировщик периодически (например, каждую секунду) опрашивает хранилище:

```
SELECT * FROM tasks WHERE execute_at <= NOW() AND status = 'pending'
```

Найденные задачи перекладываются в обычную очередь на немедленное выполнение.

**Инструменты:** Redis Sorted Set (score = timestamp выполнения), Celery Beat, Sidekiq Scheduler, AWS SQS Delay Queues (до 15 минут задержки).

#### Примеры: **Apache Airflow** (data pipelines), **Celery Beat** (Python), **Quartz** (Java), **cron** в K8s.

> Важно:  Если ваш сервис планировщик задач на нескольких инстансах (например, несколько pod в кубере) нужно использовать распределенную блокировку.
> Представим что задачи хранятся в PostgreSQL, тогда выбирать эти задачи нужно с блокировкой (select ... for update), иначе может сложиться ситуация когда несколько подов выберут и выполнят одни и те же задания.

### CDN (Content Delivery Network)

**Проблема:** Ваш сервер в Нью-Йорке. Пользователь из Токио получает картинку за 300 мс. Чем дальше пользователь от сервера, тем выше latency, и никакая оптимизация кода это не исправит: скорость света физически ограничена.

**Решение:** Географически распределённая сеть серверов, кэширующих статику (HTML, CSS, JS, изображения, видео). CDN размещает копии контента рядом с пользователем, сокращая путь данных с тысяч километров до десятков.

#### Как работает

1. Пользователь запрашивает `image.jpg`.
2. DNS направляет его на ближайший CDN-узел (PoP — Point of Presence) — по географии или по времени отклика.
3. Если файл есть в кэше узла (**cache hit**) — отдаёт сразу, origin-сервер не задействован.
4. Если файла нет (**cache miss**) — узел идёт на **Origin** (ваш сервер), забирает файл, кэширует у себя и отдаёт пользователю. Следующий запрос того же файла уже будет cache hit.

Крупные CDN (Cloudflare, Akamai) имеют сотни PoP по всему миру. Пользователь из Токио обращается к узлу в Токио, а не в Нью-Йорк — latency падает с 300 мс до 10–20 мс.

#### Что кэшировать, а что нет

CDN хорошо подходит для **статического контента** — файлов, которые одинаковы для всех пользователей и меняются редко:

- Изображения, видео, шрифты
- CSS, JS, HTML (лендинги, документация)
- Файлы для скачивания

**Не стоит кэшировать** персонализированный контент: страницу профиля пользователя, корзину, ленту новостей — они уникальны для каждого и кэш не даст эффекта, зато создаст риск отдать чужие данные.

#### Стратегии инвалидации кэша

Кэш нужно обновлять, когда исходный файл меняется. Это одна из классических проблем в Computer Science — понять, что кэш устарел.

- **TTL (Time To Live)** — файл живёт в кэше N часов, затем автоматически удаляется и при следующем запросе забирается заново с origin. Просто, но негибко: если нужно срочно обновить файл, придётся ждать истечения TTL.
- **Purge API** — ручная очистка кэша через API CDN-провайдера. Можно сбросить конкретный файл или группу файлов мгновенно. Используется при экстренных правках: опечатка на главной, битый JS-файл.
- **Версионирование файлов** (`style_v2.css`, `app.a3f9c1.js`) — **лучший подход**. Имя файла меняется при каждом изменении содержимого, поэтому браузер и CDN воспринимают его как новый файл и не используют старый кэш. TTL можно выставить бесконечным — файл с таким именем никогда не изменится. Современные сборщики (Webpack, Vite) генерируют такие имена автоматически на основе хэша содержимого.

#### CDN как защита от DDoS

CDN не только ускоряет доставку, но и выступает щитом между интернетом и вашим origin-сервером.

При DDoS-атаке тысячи запросов в секунду приходят на один адрес. Если этот адрес — ваш сервер, он падает. Если трафик идёт через CDN — атака распределяется по сотням PoP по всему миру. Каждый узел поглощает небольшую часть трафика, которую легко выдержать. Origin при этом может вообще не получить ни одного вредоносного запроса — CDN фильтрует его на периферии.

Cloudflare, например, регулярно отражает атаки мощностью свыше 1 Тбит/с именно за счёт распределённости инфраструктуры.

#### Примеры CDN провайдеров

|Провайдер|Особенность|
|---|---|
|**Cloudflare**|Бесплатный тариф, встроенная DDoS-защита, большая сеть PoP|
|**AWS CloudFront**|Глубокая интеграция с экосистемой AWS (S3, Lambda@Edge)|
|**Akamai**|Один из старейших и крупнейших, популярен в enterprise|
|**Fastly**|Мгновенная инвалидация кэша, популярен среди разработчиков|

> _Ключевая идея:_ CDN решает две задачи одновременно — снижает latency для пользователей по всему миру и защищает origin-сервер от перегрузки, поглощая как легитимный трафик, так и DDoS-атаки на периферии сети.

### База данных (БД)

**Самый сложный компонент системы.** Почти любая архитектурная проблема в конечном счёте упирается в БД: как хранить данные, как их читать достаточно быстро, как не потерять при сбое и как масштабировать при росте нагрузки. Нет "лучшей БД" — есть подходящая для конкретной задачи. Выбор БД определяется структурой данных, паттерном доступа (много чтений или много записей), требованиями к консистентности и объёмом данных.  В одном из следующих уроков мы вернемся к БД и рассмотрим более подробно такие аспекты как индексы и уровни изоляции транзакций.

#### Классификация БД

| Тип                  | Примеры            | Сценарий                               |
| -------------------- | ------------------ | -------------------------------------- |
| **Relational (SQL)** | PostgreSQL, MySQL  | Транзакции, сложные JOIN, ACID         |
| **Document**         | MongoDB, Firestore | JSON-документы, быстрое развитие схемы |
| **Key-Value**        | Redis, DynamoDB    | Кэш, сессии, рейтинги                  |
| **Wide-Column**      | Cassandra, HBase   | Огромные объёмы записей, time-series   |
| **Graph**            | Neo4j, ArangoDB    | Социальные сети, рекомендации          |

##### Relational (SQL)

Данные хранятся в таблицах со строгой схемой. Главное преимущество — **ACID-транзакции**: атомарность, консистентность, изолированность, долговечность. Это гарантирует, что перевод денег между счетами либо произойдёт полностью, либо не произойдёт вовсе. SQL-базы отлично справляются со сложными запросами с JOIN между несколькими таблицами. Подходят для большинства бизнес-приложений: банки, e-commerce, ERP-системы.

##### Document

Документы хранятся в формате JSON или BSON. Схема гибкая — разные документы в одной коллекции могут иметь разный набор полей. Удобно на ранних этапах продукта, когда схема часто меняется. Проблема возникает при сложных связях между документами: JOIN в MongoDB значительно медленнее и неудобнее, чем в SQL.

##### Key-Value

Простейшая модель: ключ → значение. Работает в памяти, поэтому невероятно быстро (Redis отвечает за микросекунды). Используется для кэша, хранения сессий, счётчиков, лидербордов. Не подходит для сложных запросов — можно получить значение только по точному ключу.

##### Wide-Column

Данные хранятся по принципу "строка → набор колонок", но колонки у разных строк могут различаться. Оптимизированы под **огромные объёмы последовательной записи**. Cassandra, например, используется для хранения логов, метрик, IoT-данных, истории сообщений. Netflix и Apple используют Cassandra для хранения сотен петабайт данных.

##### Graph

Данные представлены в виде узлов и рёбер между ними. Позволяют эффективно отвечать на вопросы вида "найди всех друзей друзей пользователя" или "какие товары часто покупают вместе". В реляционной БД такой запрос потребует множества JOIN и будет крайне медленным на больших данных. В графовой БД это нативная операция.

#### Индексы

Индекс — это отдельная структура данных (обычно B-tree), которая позволяет находить строки без полного сканирования таблицы. Запрос `WHERE user_id = 42` без индекса просматривает каждую строку таблицы. С индексом — находит результат за O(log n).

Индексы ускоряют чтение, но **замедляют запись**: при каждом INSERT/UPDATE/DELETE индекс тоже обновляется. Поэтому не стоит индексировать всё подряд — только те поля, по которым реально выполняются запросы.

#### Свойства ACID

ACID — это набор гарантий, которые реляционные БД дают при работе с транзакциями. Важно понимать эти свойства, поскольку на интервью часто спрашивают, чем SQL отличается от NoSQL.

- **Atomicity (Атомарность)** — транзакция либо выполняется целиком, либо откатывается полностью. Нет состояния "выполнена наполовину".
- **Consistency (Консистентность)** — после транзакции данные остаются в корректном состоянии (не нарушены constraints, внешние ключи и т.д.).
- **Isolation (Изолированность)** — параллельные транзакции не видят промежуточных состояний друг друга.
- **Durability (Долговечность)** — после успешного коммита данные сохранены, даже если сервер сразу упадёт.

Большинство NoSQL-баз жертвуют частью ACID-гарантий ради скорости и масштабируемости. Это осознанный трейдофф*, а не недостаток.

\* - *Трейд-офф (англ. trade-off) — это компромисс и необходимость выбора между двумя взаимоисключающими или взаимозависимыми вариантами. Подразумевает отказ от части одного блага ради получения или улучшения другого.*

#### Паттерны масштабирования

##### Master-Slave репликация

Один узел (Master) принимает все записи. Изменения асинхронно копируются на один или несколько узлов (Slave/Replica). Все запросы на чтение направляются на реплики, разгружая мастер.

Проблема: **репликационная задержка (replication lag)**. Данные появляются на реплике не мгновенно. Если пользователь записал данные и сразу же их запросил — он может получить устаревший ответ с реплики. Это называется **eventual consistency** — данные станут консистентными, но не прямо сейчас.

##### Шардирование

Горизонтальное партиционирование: данные делятся на части (шарды) и распределяются по разным серверам. Например, пользователи с `user_id % 3 == 0` хранятся на первом сервере, `== 1` — на втором, `== 2` — на третьем.

Шардирование позволяет масштабировать запись (мастер больше не один), но усложняет архитектуру: JOIN между шардами невозможен, транзакции между шардами крайне сложны, а перебалансировка шардов при добавлении нового сервера — отдельная нетривиальная задача. Шардирование применяют только тогда, когда вертикальное масштабирование и репликация уже исчерпаны.

##### CQRS (Command Query Responsibility Segregation)

Разделение на уровне архитектуры: операции записи (Command) и чтения (Query) обслуживаются разными моделями данных, а иногда и разными БД. Например, запись идёт в нормализованную PostgreSQL, а для быстрого чтения данные денормализуются и индексируются в Elasticsearch.

Подход усложняет систему, но даёт гибкость: можно независимо масштабировать чтение и запись, оптимизировать каждую модель под свой паттерн доступа.

#### Как выбирать БД на интервью

Универсальный ориентир при проектировании системы:

- Нужны транзакции и сложные связи между сущностями → **PostgreSQL**
- Нужен быстрый кэш или счётчики → **Redis**
- Огромные объёмы записей, time-series, логи → **Cassandra**
- Полнотекстовый поиск → **Elasticsearch**
- Связи между объектами (социальный граф) → **Neo4j**

На практике большинство систем используют **несколько БД одновременно**: PostgreSQL как основное хранилище, Redis для кэша и сессий, Elasticsearch для поиска. Каждая БД решает ту задачу, для которой оптимизирована.

> _Ключевая идея:_ Нормализация для записи (меньше дублирования, проще поддерживать консистентность), денормализация для чтения (меньше JOIN, быстрее запросы). Это противоречие и является главной причиной, почему в высоконагруженных системах используют несколько разных БД одновременно. Подробно паттерны работы с БД, шардирование и репликацию разберём в отдельном уроке.

### Кэш (Cache)

**Кэш** — это быстрое промежуточное хранилище, которое сохраняет результаты дорогостоящих операций (запросов к БД, вычислений, сетевых запросов), чтобы отдавать их без повторного выполнения этих операций. Подробнее мы разберем кэш в уроке 4.

**Проблема:** БД — узкое место. Чтение с диска (даже SSD) в 1000 раз медленнее RAM.

**Решение:** Временное хранилище в оперативной памяти.

#### Стратегии:

- **Cache-Aside (ленивая загрузка):** Сначала смотрим в кэш, если нет — берем из БД и кладем в кэш. Самый популярный.
- **Write-Through:** Сначала пишем в БД, потом обновляем кэш.
- **Write-Behind:** Пишем только в кэш, он сам асинхронно сливает в БД (риск потерять данные).

#### Проблемы:

- **Thundering Herd:** Тысячи запросов к одному ключу после инвалидации. Решения:        **1) Блокировка**: Когда происходит cache miss, только **один** запрос получает право идти в БД. Остальные ждут, пока кэш не прогреется.  **2) Ранее перестроение** - основная идея: **не ждать истечения TTL**, а обновлять кэш заранее, пока старое значение ещё актуально.
- **Cache Penetration:** Запрос к несуществующим данным (всегда летит в БД). Решение — кэшировать `null`.
- **Cache Avalanche:** Массовый сброс кэша из-за одинакового TTL. Решение — TTL с разбросом (+ случайные секунды)

#### In memory cache
Часто, чтобы не  "тащить"  на проект дополнительную БД для кэша (например redis) можно "прикрутить" in memory cache прямо к сервису.  Для этого есть готовые пакеты на го. Но здесь стоит помнить о проблеме синхронизации данных между инстансами, если сервис крутится в кубере. Иногда иметь копию кэша на каждом инстансе оказывается вполне рабочим вариантом.

#### Технологии: **Redis**, **Memcached**, **KeyDB** (форк редис), **Hazelcast** (чаще в Java экосистеме).

### Очереди сообщений (Message Queues)

**Проблема:** Синхронный запрос держит пользователя в ожидании, пока бэкенд выполняет тяжёлую работу — отправку 10 писем, ресайз видео, генерацию PDF-отчёта. Пользователь ждёт 30 секунд, получает таймаут, жмёт кнопку ещё раз — задача запускается дважды. Это плохой UX и ненадёжная архитектура.

**Решение:** Асинхронная обработка через брокер сообщений. Бэкенд мгновенно кладёт задачу в очередь и отвечает пользователю "принято". Воркер забирает задачу из очереди и выполняет её в фоне. Пользователь не ждёт — он получает уведомление когда работа завершена.

#### Модели доставки

**Queue (Point-to-Point):** Одно сообщение → один консьюмер. Сообщение удаляется из очереди после обработки. Если консьюмеров несколько — каждое сообщение достаётся только одному из них (балансировка нагрузки). Пример: очередь на отправку email. Не нужно, чтобы письмо отправляли пять воркеров одновременно.

**Pub/Sub (Topic):** Одно сообщение → много консьюмеров одновременно. Продюсер публикует событие в топик, все подписчики получают копию. Пример: событие "пользователь зарегистрировался" → (1) сервис нотификаций отправляет приветственное письмо, (2) сервис бонусов начисляет баллы, (3) аналитика фиксирует конверсию. Продюсер ничего не знает о подписчиках — слабая связанность между сервисами.

#### Ключевые термины

**Продюсер (Producer)** — сервис, который отправляет сообщения в брокер.

**Консьюмер (Consumer)** — сервис, который читает и обрабатывает сообщения из брокера.

**Брокер (Broker)** — посредник между продюсером и консьюмером. Принимает, хранит и доставляет сообщения. Гарантирует, что сообщение не потеряется, даже если консьюмер временно недоступен.

**Acknowledgement (ACK)** — подтверждение от консьюмера, что сообщение успешно обработано. Пока брокер не получил ACK, он считает сообщение необработанным. Если консьюмер упал — брокер отдаст сообщение другому консьюмеру.

**Consumer Group** — группа консьюмеров, которые совместно читают из одного топика, разделяя нагрузку между собой. Kafka распределяет партиции топика между консьюмерами в группе: каждую партицию читает ровно один консьюмер из группы.

#### Свойства и гарантии

**At-least-once + Idempotency** — стандарт индустрии. Брокер гарантирует, что сообщение будет доставлено хотя бы раз. При сбое сообщение будет доставлено повторно, поэтому консьюмер должен быть идемпотентным — повторная обработка не должна ломать логику.

**Dead Letter Queue (DLQ)** — отдельная очередь для сообщений, которые не удалось обработать за N попыток. Вместо того чтобы блокировать основную очередь бесконечными ретраями, "ядовитое" сообщение отправляется в DLQ для ручного анализа. Причины могут быть разные: невалидные данные, баг в коде консьюмера, недоступная внешняя система.

**Backpressure** — механизм защиты от перегрузки. Если консьюмеры не справляются с темпом продюсеров, очередь растёт. Backpressure сигнализирует продюсеру замедлиться, чтобы система не накопила миллионы необработанных сообщений и не исчерпала память.

#### Примеры брокеров

**RabbitMQ** — классический брокер с умной маршрутизацией. Поддерживает сложные правила: отправить сообщение в разные очереди в зависимости от его содержимого (routing key). Сообщения удаляются после обработки — RabbitMQ не предназначен для долгосрочного хранения событий. Хорошо подходит для task queue в микросервисах.

**Apache Kafka** — лог-ориентированный брокер. Принципиальное отличие от RabbitMQ: сообщения **не удаляются** после обработки, они хранятся на диске днями или неделями. Консьюмер сам отслеживает, до какого места он дочитал (offset). Это позволяет: воспроизводить события заново, добавлять новых консьюмеров которые прочитают всю историю, восстанавливаться после сбоя. Kafka оптимизирована под огромный throughput — миллионы сообщений в секунду. Используется как шина событий в крупных системах (LinkedIn, Uber, Airbnb).

**AWS SQS** — облачный (fully managed) сервис от Amazon. Не нужно разворачивать и поддерживать инфраструктуру брокера. Поддерживает стандартные очереди (at-least-once) и FIFO-очереди (exactly-once, гарантированный порядок). Легко интегрируется с другими AWS-сервисами.

#### Важный паттерн: Request-Reply с очередями

Обычно очереди используются для одностороннего fire-and-forget: продюсер отправил и забыл. Но иногда нужно получить результат асинхронной операции.

Клиент кладёт задачу в `queue.requests` вместе со своим идентификатором. Консьюмер обрабатывает задачу и кладёт результат в `queue.responses:{clientId}`. Клиент подписан на свою персональную очередь ответов и забирает результат когда он появился.

```
Клиент → [queue.requests] → Воркер
                                ↓
Клиент ← [queue.responses:client-42] ← Воркер
```

Этот паттерн позволяет сохранить асинхронность и при этом вернуть результат конкретному клиенту — например, при генерации отчёта или долгом вычислении.

#### Когда использовать брокер сообщений

|Сценарий|Почему брокер|
|---|---|
|Тяжёлые фоновые задачи (ресайз, PDF)|Не блокировать HTTP-запрос пользователя|
|Коммуникация между микросервисами|Слабая связанность, сервисы не знают друг о друге|
|Выравнивание нагрузки|Очередь поглощает пики, воркеры работают в своём темпе|
|Гарантированная доставка|Брокер хранит сообщение пока консьюмер не подтвердит обработку|
|Аудит и воспроизведение событий|Kafka хранит историю — можно перечитать события заново|

> _Ключевая идея:_ Брокер сообщений decouples (разъединяет) продюсера и консьюмера в пространстве (они не знают адресов друг друга), во времени (консьюмер может быть недоступен в момент отправки) и в скорости (они работают в разном темпе). Именно это делает брокер одним из ключевых строительных блоков масштабируемых систем.

### Заключение: Пример как все компоненты могут работать вместе?

Рассмотрим **запрос пользователя**:

1. **CDN** → отдал статику (HTML, логотип).
2. **Балансировщик (L7)** → перенаправил запрос `/api/order` на микросервис "корзина".
3. **Rate limiter** → проверил, не превысил ли пользователь лимит (200/мин).
4. Микросервис **проверил кэш (Redis)** → там нет корзины.
5. Достал из **SQL БД** → закэшировал обратно.
6. Если нужно отправить уведомление → положил задачу в **RabbitMQ**.
7. **Job scheduler** раз в 5 минут собирает статистику заказов.
8. **Генератор ID (Snowflake)** выдал уникальный номер для заказа.

Из данных компонентов можно проектировать различные системы. Мы вернемся к ним в уроке 8, когда будем проектировать ленту новостей.
