### Балансировщик нагрузки (load balancer)

**Проблема:** Один сервер не справляется с трафиком. Если он упадет — система умрет.
**Решение:** Распределение входящих запросов между несколькими серверами (бэкендами).

#### Типы:
- **L4 (Транспортный уровень):** Работает с TCP/UDP, не смотрит на содержимое запроса. Быстро, но "слепой" к HTTP-методам.
- **L7 (Прикладной уровень):** Понимает HTTP/HTTPS. Может маршрутизировать запросы на основе URL (например, `/api/*` → одни серверы, `/static/*` → другие).

#### Основные алгоритмы:

- Round Robin (самый простой и популярный: по кругу)
- Weighted RR (по кругу с весами)
- Least Connections (куда меньше занято)
- Least Response Time (где наименьшее время ответа)
- IP Hash (для привязки сессий, sticky session)

##### Round Robin 
**Механизм:** Балансировщик хранит указатель на последнего выбранного сервера. При каждом новом запросе указатель сдвигается на следующего по списку сервера. Дойдя до конца списка, возвращается к началу.

**Как это работает:** Запросы распределяются строго по очереди: 1-й → сервер A, 2-й → сервер B, 3-й → сервер C, 4-й → снова A.

##### Weighted RR
**Механизм:** Каждому серверу присваивается вес (например, A — вес 5, B — вес 1).  Обычно вес зависит от "железа" сервера: больший вес присваивается более мощному серверу. Балансировщик составляет виртуальную последовательность, где сервер A повторяется 5 раз, а сервер B — 1 раз. Затем применяет обычный Round Robin к этой последовательности.

**Как это работает:** Если сервер мощнее — он получает пропорционально больше запросов. При весах 5 и 1 на каждые 6 запросов первый сервер обработает 5, второй — 1.

##### Least Connections
**Механизм:** Запрос направляется серверу, у которого в данный момент меньше всего активных соединений.
 
**Как работает:** Балансировщик ведёт счётчик открытых соединений на каждом сервере.
 
**Идеально для:** Долгих сессий (например, загрузка файлов, потоковое видео, API с разным временем ответа).

##### Least Response Time
**Механизм:** Отправляет запрос серверу, который одновременно имеет меньше всего активных соединений и наименьшее среднее время ответа.

**Как работает:** Балансировщик хранит и постоянно перерассчитывает экспоненциально взвешенное скользящее среднее время ответа и отправляет запрос на сервер, у которого оно наименьшее.

**Минус:** Требует постоянного мониторинга задержек, что создаёт небольшие накладные расходы.

##### IP Hash
**Механизм:** Балансировщик вычисляет хэш от IP-адреса клиента (или его id) и направляет запрос на определённый по этому хэшу сервер.

**Как работает:** Балансировщик вычисляет хэш от IP-адреса клиента (например, CRC32 или MD5), затем берёт остаток от деления этого хэша на количество серверов в пуле. Полученное число определяет, какой сервер из упорядоченного списка получит запрос. Поскольку хэш от одного и того же IP всегда одинаков, а список серверов не меняется, все запросы одного клиента попадают на один и тот же сервер.

**Цель:** Обеспечить «липкость» (sticky session) — чтобы запросы одного пользователя всегда попадали на один и тот же сервер (нужно для хранения сессий в локальной памяти).

**Проблема:** При добавлении/удалении сервера хэш-распределение ломается (проблема решается консистентным хэшированием, о котором мы поговорим в отдельном уроке). 


#### Другие алгоритмы:
Существуют и другие более экзотически алгоритмы балансировки нагрузки, например Dynamic Load Balancing или Power of Two Choices. Их крайне редко спрашивают на интервью, поэтому не будем на них останавливаться. Если интересно, можно почитать про них самостоятельно.

#### Примеры балансировщиков:
* **Nginx**
* HAProxy, 
* AWS ELB, 
* Istio

> _Ключевая идея:_ Балансировщик должен сам быть отказоустойчивым (Active-Passive или Active-Active кластер).

### Rate Limiter (Лимитер запросов)

**Проблема:** Один клиент (или злоумышленник) может положить систему DDoS-атакой или просто багом в коде, отправляя миллионы запросов в секунду.

**Решение:** Ограничение количества запросов от одного пользователя/IP за интервал времени.

#### Алгоритмы (стратегии):

1. **Token Bucket:** Самый популярный. В ведро каждую секунду падает N токенов. Запрос забирает токен. Нет токена — ответ `429 Too Many Requests`.
    
2. **Leaky Bucket:** Запросы попадают в очередь и обрабатываются с фиксированной скоростью.
    
3. **Sliding Window Log:** Хранит временные метки запросов. Точно, но затратно по памяти.
    

#### Где размещать:

- На стороне клиента (ненадежно).
- **На стороне балансировщика или API Gateway** (лучше всего).
- Как отдельный сервис (Redis + Lua скрипты).

> _Ключевая идея:_ Rate limiter должен быть распределенным — если у вас 10 балансировщиков, они должны синхронизировать счетчики через Redis.

###  Генератор ID (ID Generator)

**Проблема:** В распределенной системе нельзя просто взять `AUTO_INCREMENT` из MySQL — тормоза и single point of failure.

**Требования к ID:** Глобальная уникальность, сортируемость по времени (для индексов B-Tree), компактность.

### Решения:

|Подход|Описание|Плюсы / Минусы|
|---|---|---|
|**UUID v4**|128 бит случайности|Просто, но не сортируемый, большой размер|
|**Snowflake (Twitter)**|64 бита: timestamp + машинный ID + счетчик|**Стандарт индустрии**. Компактный, сортируемый|
|**Leaf (Meituan)**|Выделяет сегменты ID (например, 1-1000 серверу А)|Уменьшает обращения к БД|

### Формат Snowflake (41 бит времени + 10 бит ноды + 12 бит seq):

Максимум 4096 ID в миллисекунду на одну ноду.

> _Ключевая идея:_ Генератор ID — типичный **функциональный сервис** без состояния или с легким состоянием (ZooKeeper для координации нод).

### Job Scheduler (Планировщик заданий)

**Проблема:** Нужно выполнять фоновые задачи: отправить email в 10 утра, ресайзить картинки раз в час, чистить логи.

**Решение:** Система для запуска задач по расписанию или по событию.

#### Архитектура:

- **Master-Worker:** Мастер хранит расписание в БД (или etcd), выдает задания воркерам.
    
- **Leader Election:** Несколько нод, одна — лидер (через ZooKeeper/Consul).
    

#### Важные концепты:

- **At-most-once** (почта) vs **At-least-once** (генерация отчета).
    
- **Idempotency** — задача должна быть безопасна при повторном запуске.
    
- **Delayed Queue** — если задачу нужно выполнить "через 30 минут".
    

#### Примеры: **Apache Airflow** (data pipelines), **Celery Beat** (Python), **Quartz** (Java), **cron** в K8s.

> _Ключевая идея:_ Не храните шедулер на одном сервере — используйте распределенную блокировку.

### CDN (Content Delivery Network)

**Проблема:** Ваш сервер в Нью-Йорке. Пользователь из Токио получает картинку за 300 мс.

**Решение:** Географически распределенная сеть серверов, кэширующих статику (HTML, CSS, JS, изображения, видео).

#### Как работает:

1. Пользователь запрашивает `image.jpg`.
    
2. DNS направляет его на ближайший CDN-узел (PoP — Point of Presence).
    
3. Если у узла нет файла — он идет в **Origin** (ваш сервер), кэширует и отдает.
    

#### Стратегии инвалидации кэша:

- TTL (время жизни — например, 24 часа).
    
- Purge API (ручная очистка).
    
- Версионирование файлов (`style_v2.css`) — лучший подход.
    

#### Примеры: **CloudFlare**, **Akamai**, **AWS CloudFront**, **Fastly**.

> _Ключевая идея:_ CDN не только ускоряет, но и защищает от DDoS — атака гасится на периферии.

### База данных (БД)

**Самый сложный компонент.** Нет "лучшей БД", есть подходящая для задачи.

### Классификация:

|Тип|Примеры|Сценарий|
|---|---|---|
|**Relational (SQL)**|PostgreSQL, MySQL|Транзакции, сложные JOIN, ACID|
|**Document**|MongoDB, Firestore|JSON-документы, быстрое развитие схемы|
|**Key-Value**|Redis, DynamoDB|Кэш, сессии, рейтинги|
|**Wide-Column**|Cassandra, HBase|Огромные объемы записей, time-series|
|**Graph**|Neo4j, ArangoDB|Социальные сети, рекомендации|

### Паттерны масштабирования:

- **Master-Slave репликация** (читаем с реплик, пишем в мастер).
    
- **Шардирование** (горизонтальное партиционирование по ключу, например `user_id % 4`).
    
- **CQRS** (отдельные БД на запись и чтение).
    

> _Ключевая идея:_ Нормализация для записи, денормализация для чтения.

### Кэш (Cache)

**Проблема:** БД — узкое место. Чтение с диска (даже SSD) в 1000 раз медленнее RAM.

**Решение:** Временное хранилище в оперативной памяти.

#### Стратегии:

- **Cache-Aside (ленивая загрузка):** Сначала смотрим в кэш, если нет — берем из БД и кладем в кэш. Самый популярный.
    
- **Write-Through:** Сначала пишем в БД, потом обновляем кэш.
    
- **Write-Behind:** Пишем только в кэш, он сам асинхронно сливает в БД (риск потерять данные).
    

#### Проблемы:

- **Thundering Herd:** Тысячи запросов к одному ключу после инвалидации. Решение — блокировка или "раннее перестроение".
    
- **Cache Penetration:** Запрос к несуществующим данным (всегда летит в БД). Решение — кэшировать `null`.
    
- **Cache Avalanche:** Массовый сброс кэша из-за одинакового TTL. Решение — TTL с разбросом (+ случайные секунды).
    

#### Технологии: **Redis** (король кэша), **Memcached** (проще, без репликации).

### Очереди сообщений (Message Queues)

**Проблема:** Синхронный запрос делает пользователя слишком долго, если бэкенд выполняет тяжелую работу (отправка 10 писем, ресайз видео).

**Решение:** Асинхронная обработка через брокер сообщений.

#### Модели:

- **Queue (Point-to-Point):** Одно сообщение → один консьюмер. Пример: отправка email.
    
- **Pub/Sub (Topic):** Одно сообщение → много консьюмеров. Пример: событие "пользователь зарегистрировался" → (1) приветственное письмо, (2) начисление бонусов, (3) аналитика.
    

#### Свойства:

- **At least once** + **Idempotency** — стандарт индустрии.
    
- **Dead Letter Queue (DLQ):** Сообщения, которые не удалось обработать за N попыток, улетают сюда для ручного анализа.
    

#### Примеры:

- **RabbitMQ** (умный брокер, сложная маршрутизация).
    
- **Apache Kafka** (лог-ориентированная, огромный throughput, хранение событий дни/недели).
    
- **AWS SQS** (полностью managed).
    

#### Важный паттерн: **Request-Reply с очередями**

Клиент кладет задачу в `queue.requests`, консьюмер кладет результат в `queue.responses:{clientId}`.

### Заключение: Пример как все компоненты могут работать вместе?

Рассмотрим **запрос пользователя**:

1. **CDN** → отдал статику (HTML, логотип).
    
2. **Балансировщик (L7)** → перенаправил запрос `/api/order` на микросервис "корзина".
    
3. **Rate limiter** → проверил, не превысил ли пользователь лимит (200/мин).
    
4. Микросервис **проверил кэш (Redis)** → там нет корзины.
    
5. Достал из **SQL БД** → закэшировал обратно.
    
6. Если нужно отправить уведомление → положил задачу в **RabbitMQ**.
    
7. **Job scheduler** раз в 5 минут собирает статистику заказов.
    
8. **Генератор ID (Snowflake)** выдал уникальный номер для заказа.
